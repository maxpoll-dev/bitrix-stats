## Что это

Take-home задание с лимитом 6–8 часов: построить первый аналитический контур поверх выгрузки из CRM (похожей на Bitrix24) и производственных таблиц. Целевой пайплайн:

```
CSV-выгрузка → PostgreSQL (raw/staging слой) → нормализованный слой → SQL-отчеты → ERD + BPM/FSM-модели + план dashboard/API
```

Полное ТЗ — в `TASK.md`. Рабочий язык документации/коммитов — русский.

## Исходные данные (`data/*.csv`)

13 таблиц, маленькие фикстуры (по 3–12 строк), специально подобранные для проверки навыков моделирования и работы с качеством данных. Ключевые связи:

- `deals` — центральная таблица. FK: `stage_id`→`pipeline_stages`, `manager_id`→`users`, `company_id`→`companies`, `contact_id`→`contacts`.
- `deal_products` (связь deal↔product M:N с qty/price/discount), `payments`, `activities`, `stage_history`, `production_orders`, `shipments` — все ссылаются на `deal_id`.
- `contacts.company_id`→`companies`. `marketing_costs` связывается со сделками только по `source` + дате (без FK).
- `pipeline_stages` задаёт порядок стадий и флаги финальности/успеха (`is_final`, `is_success`).

## Намеренные дефекты качества данных (НЕ «чинить» молча в raw-слое)

В фикстуры заложены проблемы, которые по заданию нужно обнаружить и описать в `DATA_QUALITY.md`, а затем обработать в нормализованном слое:

- **Дубль `deal_id` D1008** (две конфликтующие строки — нужна стратегия дедупликации).
- **Отрицательный `expected_amount`** (D1007).
- **Разнобой в `source` по регистру/написанию**: `Avito`/`avito`/`AVITO`/`Авито`, `website`/`Website` — нужна нормализация в справочник источников перед джойном с `marketing_costs`.
- **Неизвестная стадия** `WAIT_CLIENT` (D1011), которой нет в `pipeline_stages` (битый FK).
- **Разный формат даты**: у D1010 `created_at` = `14.06.2026 09:20`, тогда как у остальных ISO-8601 со смещением `+05:00`.
- **Пропуски**: у D1010 пустой `currency`; у D1011 null `manager_id`; сделки с null `company_id`/`contact_id`; сделки `WON` без `closed_at` (D1009).

Сохраняй сырые данные как есть; чистку/типизацию/дедуп делай на трансформации staging→нормализованный слой, а каждое допущение фиксируй в `ASSUMPTIONS.md`.

## Требуемые отчёты (`reports.sql`)

Текущая воронка (количество + сумма по стадиям); продажи и маржа по менеджерам; дебиторка (сумма сделки − оплаты = остаток); сделки с задержкой производства >5 дней; сделки без активности за последние N дней; источники заявок с выручкой/маржой; сводный запрос по проблемам данных.
