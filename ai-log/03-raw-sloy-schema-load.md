# 03 — Raw-слой: 01_schema.sql и 02_load.sql

## Промпт
> Создай два файла в db/init. 01_schema.sql - все значения строки. 02_load.sql - импорт таблиц из папки data с сохранением названий
>
> задокументируй

## Ответ ИИ
> Созданы два файла:
>
> **`db/init/01_schema.sql`** — raw-слой. Схема `raw`, 13 таблиц по числу CSV, все столбцы `text`, без PK и без ограничений. Причина: загрузка не должна падать на дублях (D1008), битых ссылках (D9999) и кривых форматах дат/чисел — всё это разбирается на следующем слое. Имена таблиц и столбцов совпадают с файлами и заголовками CSV. Каждая таблица с `DROP TABLE IF EXISTS` для идемпотентности.
>
> **`db/init/02_load.sql`** — импорт через `COPY ... FROM '/data/<файл>.csv' WITH (FORMAT csv, HEADER true)` для всех 13 таблиц.
>
> **Допущение:** папка `./data` монтируется в контейнер как `/data` (docker-compose `volumes: ./data:/data:ro`). `COPY` выполняется на стороне сервера, поэтому файлы должны быть доступны изнутри контейнера. Если монтировать в другой путь — поправить пути в `02_load.sql`. Зафиксировать монтирование при создании docker-compose.
>
> Файлы лежат в `db/init/` с числовыми префиксами — это стандартный порядок выполнения для `docker-entrypoint-initdb.d`.
